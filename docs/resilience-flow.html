Each request passes through three nested pipelines. Retry is the outermost — it can re-execute everything inside it. Circuit Breaker sits in the middle and can short-circuit the whole attempt. Timeout is the innermost, applied per attempt.

① Outermost RETRY
RetryPipeline
Re-executes everything inside on: HttpRequestException, TaskCanceledException, HTTP 408, 429, 5xx.
Delay uses exponential back-off — 500ms, 1000ms, 2000ms… (base from RetryDelayMilliseconds).
Max attempts from Retry config or per-request override.
↓ if no transient error, passes through; on error → waits and re-enters
② Middle CIRCUIT BREAKER
CircuitBreakerPipeline
Shared per (concrete ExternalServiceConfig type, response type T) — same config class with different GetAsync<T> types uses separate breaker state.
Trips when failure ratio ≥ FailureRatio within SamplingDurationSeconds after MinimumThroughput requests.
While open: no HTTP call; HttpClientService catches Polly’s BrokenCircuitException and returns HttpApiResponse<T> with synthetic 503.
↓ if circuit is closed → proceeds; if open → structured error response (no network)
③ Innermost — per attempt TIMEOUT
TimeoutPipeline
Cancels the HTTP call after TimeoutSeconds (config default) or HttpRequestParameter.Timeout (per-request override).
Fires independently of the outer CancellationToken — each attempt gets its own deadline.
↓ sends the real HTTP request
④ Actual call HTTP
BuildAndSendRequestAsync
Builds HttpRequestMessage with resolved URI, headers, bearer token, correlation ID.
Calls HttpClient.SendAsync → reads response → deserializes to T.
Key insight: because Retry is outermost, each retry attempt re-enters the Circuit Breaker. This means a series of failures caused by retries also contributes to the circuit breaker's failure count — which is exactly what trips it open, protecting the downstream service from being hammered.
✓ Scenario A Happy path — first attempt succeeds
⚠ Scenario B First attempt fails (503) — retry succeeds
✕ Scenario C Circuit breaker is OPEN — call short-circuits immediately
CLOSED
Normal operation.
Requests pass through.
Failures are counted.
OPEN
Service is failing.
All requests blocked immediately.
No network calls.
HALF-OPEN
Recovery probe.
One trial request allowed.
Pass → Close, Fail → Open.
CLOSED → OPEN Failure ratio ≥ FailureRatio (default 0.5) within the last SamplingDurationSeconds (default 30s), after at least MinimumThroughput (default 10) requests have been made.
OPEN → HALF-OPEN BreakDurationSeconds (default 30s) have elapsed since the circuit opened. One trial request is allowed through.
HALF-OPEN → CLOSED The trial request succeeds. Circuit resets, failure counters cleared.
HALF-OPEN → OPEN The trial request fails. Circuit opens again for another BreakDurationSeconds.
Circuit scope: breaker state is keyed by (concrete ExternalServiceConfig type, typeof(T)). Different config classes (e.g. PaymentApiConfig vs NotificationApiConfig) are isolated; the same config with different response types also gets separate breakers.

Named HttpClient scope: each config subclass also gets its own isolated HttpClient registration, keyed by ClientName (defaults to the class name). You can override it directly in appsettings.json ("ClientName": "testeapi123") or in the constructor (ClientName = "testeapi123") — no code subclassing required.

Every method returns HttpApiResponse<T>. Pipeline-level failures (network after retries, circuit open, timeout after retries, bad URI configuration) are returned as structured responses — not thrown to your code. Exception: System.Text.Json deserialization of the response body can still throw if the payload is invalid for T.

USAGE EXAMPLE
var result = await httpClient.GetAsync<MyResponse>(new HttpRequestParameter { Path = "..." }, ct);

// Infrastructure error (network, circuit breaker, timeout, misconfiguration)
if (result.ErrorMessage is not null)
{
    logger.LogError("Call failed: {Code} — {Msg}", result.StatusCode, result.ErrorMessage);
    return null;
}

// Normal HTTP error (4xx / 5xx)
if (!result.IsSuccessStatusCode)
{
    logger.LogWarning("HTTP {Code}: {Reason}", result.StatusCode, result.ReasonPhrase);
    return null;
}

// Success
return result.Content;

A comparison of what you'd need to implement manually versus what the library provides out of the box.

🔁
Zero Polly boilerplate
Setting up Polly v8 correctly requires ResiliencePipelineBuilder, retry predicates, circuit breaker options, timeout chaining, and correct nesting order. This lib does all of it for you — configured from appsettings.json or inline objects with no boilerplate.
⚡
Isolated circuit breakers
State lives in a static ConcurrentDictionary keyed by (concrete config type, response type). Different APIs (PaymentApiConfig vs NotificationApiConfig) do not share breakers; different T on the same config also splits state.
🔗
Correlation ID propagation
Every outbound request automatically carries a correlation ID header (X-Correlation-Id by default). You can supply your own from an upstream request or let the library auto-generate a GUID. Configurable header name and GenerateIfAbsent behaviour per service.
📦
Typed response envelope
Every call returns HttpApiResponse<T> with the HTTP status from the server for typical responses. You always get IsSuccessStatusCode, StatusCode, ReasonPhrase and the deserialized Content in one object. Fewer try/catch blocks for network, breaker, timeout, and config errors (returned in the envelope).
🔒
Safe debug logging out of the box
Enable EnableDebugLogging = true per request and get a full curl command + sanitized response body at Debug level. Sensitive fields (password, token, apikey, email, ssn, …) are automatically redacted — no risk of leaking secrets to logs.
🎛️
Two-tier configuration
Service-wide defaults are set once in config (Retry = 3, TimeoutSeconds = 10). Any individual request can override them via HttpRequestParameter.Retry or .Timeout — without touching the service config or creating a new service instance.
🏭
IHttpClientFactory friendly — isolated per service
Uses a named IHttpClientFactory internally — the correct way to manage HttpClient lifetime in .NET. Avoids socket exhaustion and DNS staleness.

The name comes from ClientName (defaults to the config class name, e.g. "PaymentApiConfig"), so each service gets its own independent connection pool and BaseAddress. To use a custom name, simply set "ClientName": "myname" in appsettings.json or assign it in the constructor — no subclass override required.
📍
Path + BaseUri — no URL repetition
Set BaseUri once in config. Each call passes only the relative Path ("/users/123"). For cross-service calls, override with an absolute Uri. The library resolves the final URL automatically, with clear error messages if misconfigured.
